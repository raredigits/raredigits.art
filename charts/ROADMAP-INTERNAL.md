> **Про нумерацию 0.9.8_x и CDN.** Тег вида `v0.9.8_1` — невалидный semver (как исторические `v0.9.4_1…_4`): jsDelivr его игнорирует при резолве `@latest`, который останется на `v0.9.8` до выхода 0.9.9. Пиновые URL (`@v0.9.8_1`) работают нормально. То есть промежуточные релизы — **opt-in по пину**: боевые проекты подключают их явно, случайные потребители `@latest` не затронуты. Релизный гейт и автотег с такими версиями совместимы (сверка идёт по точному имени тега).

> **Риск, о котором помним:** классы из 0.9.8_x замораживаются в 1.0 молодыми. Митигция: выпускать релизы по мере готовности (не пачкой перед 0.9.9), сразу заводить каждый новый класс в [API-INVENTORY](./API-INVENTORY-INTERNAL.md) и прогонять через дашборд-проект до заморозки.

Пункты с пометкой **[parallel]** — дешёвые и независимые, можно делать в любой момент.

## 1.0 = заморозка API

Главный смысл мажорной версии по semver: публичная поверхность (имена опций, форма данных, имена классов) замораживается, и внешние проекты могут на неё полагаться. Библиотека уже грузится по CDN на боевых проектах — фактически контракт уже действует, но формально он ещё Beta. Всё в этом файле — про то, чтобы к 1.0 этот контракт был надёжным и проверяемым.

Решение 2026-07-06: заморозка фиксирует **полный терминальный словарь** — недостающие жанры (composition, scatter/heatmap/waterfall, шкалы) доезжают релизами 0.9.8_x, и 1.0 обещает стабильность уже для них тоже. Цена решения — молодым классам нужна обкатка до заморозки (см. риск-заметку под планом релизов); 0.9.9 существует ровно для этой финальной отладки.

## Блок R. Релизная инфраструктура (заведён 2026-07-03)

### R1. Тесты не гоняются в CI — ✅ сделано 2026-07-03
**Где:** `.github/workflows/{deploy,sync-charts}.yml` — ставили зависимости, собирали, но `npm test` не запускали.
**Решение:** шаг `Run tests` (`npm run test:run`) добавлен в оба воркфлоу перед сборкой; падение тестов = нет деплоя/синка.

### R2. CDN получает незарелиженный код — ✅ сделано 2026-07-03 (проверить на релизе 0.9.8)
**Где:** `sync-charts.yml` триггерится на push в `assets/charts/src/**` и пушит свежесобранные бандлы в `raredigits/rare-charts@main`.
**Уточнение (2026-07-03):** в `rare-charts` **есть semver-теги** (v0.9.0…v0.9.7), и jsDelivr резолвит `@latest` по тегам, а не по `main`. Так что потребители `@latest` и пиновых версий защищены — незарелиженный код на `main` их не задевает. Но: теги ставятся **руками** и не упомянуты в release-чеклисте CHANGELOG (скрытый шаг); `main` — де-факто канарейка, из которой непонятно, что было релизом; попадаются кривые теги (`v.0.9.2`, `v0.9.4_1…_4` — невалидный semver, jsDelivr их игнорирует).
**Решение (реализовано 2026-07-03):** синк стал релизным — первый шаг воркфлоу сверяет версию из `version.json` с тегами `rare-charts` (`git ls-remote --tags`): тег есть → все шаги скипаются; тега нет → тесты, сборка, синк, коммит `RareCharts vX.Y.Z` и пуш **вместе с тегом**. Ручное тегирование ушло, `main` в `rare-charts` теперь состоит только из релизных коммитов. Релиз = бамп `version.json` + push. ⚠️ Проверить на живом релизе 0.9.8 — первый прогон гейта.

### R3. Бандлы в git отстают от src — ✅ сделано 2026-07-03
**Где:** `assets/charts/rare-charts{.js,.min.js}` — в git лежала сборка v0.9.7 без `[Unreleased]`-изменений.
**Решение:** бандлы убраны из git (`git rm --cached` + `.gitignore`) — их никто не потреблял из репозитория (деплой, CDN-синк и локальный dev пересобирают из `src/` сами), а устареть они могли. Источник правды в git — только `src/`; готовые файлы раздаются из `raredigits/rare-charts`. Рассинхрон теперь невозможен по построению. Release-чеклист в CHANGELOG обновлён (шаг «пересобрать бандлы» заменён на «CI соберёт сам», ручное тегирование `rare-charts` записано явно до автоматизации R2).
